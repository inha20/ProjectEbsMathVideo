# MathVideo AI - 기술 면접 Q&A 대비

이 문서는 이 프로젝트를 기반으로 한 기술 면접에서 나올 수 있는 핵심 질문과 모범 답변을 정리한 것입니다.

## 1. 아키텍처 패턴
**Q1: 이 프로젝트는 어떤 아키텍처 패턴을 따르고 있나요? 그 이유는 무엇인가요?**
**답변:**
이 프로젝트는 Controller-Service-Repository 패턴 기반의 계층형 아키텍처(Layered Architecture)를 따르고 있습니다. 명확한 분리를 통해 유지보수성과 테스트 용이성을 높이기 위함입니다. `Dependency Injection`(의존성 주입)을 통해 Controller는 Service 인터페이스에만 의존하게 하여 커플링을 낮췄습니다. 기존 `main.py`에 집중되어 있던 라우터 로직을 분리해 모듈로서의 확장성을 얻었습니다.

## 2. 데이터베이스 모델링
**Q2: 시스템에서 사용되는 핵심 DB 모델의 구조와 RDBMS를 선택한 이유는?**
**답변:**
핵심 테이블은 ChatSession, ChatMessage, Playlist 등입니다. 관계 데이터 모델링을 통해 세션과 메시지가 1:N 관계로 묶입니다. SQLite를 사용했지만 추후 PostgreSQL로 전환을 고려하여 SQLAlchemy의 ORM 기능을 활용해 구현했습니다.

## 3. 추천 알고리즘과 한계
**Q3: 현재 EBS 강좌 추천의 알고리즘은 무엇이며, 개선 방향은?**
**답변:**
현재 EBS 추천은 Rule-based Scoring 알고리즘입니다. (학년 10점, 토픽별 5점 등) 사용자의 입력에서 키워드를 추출해 점수를 매깁니다. 이의 한계점은 오타나 동의어 처리에 취약하다는 점입니다. 이를 개선하기 위해 TF-IDF 가중치를 적용하거나, 한국어 형태소 분석기를 적용해 검색 유연성을 더하는 것이 개선 과제입니다.

## 4. YouTube API 활용과 안정성
**Q4: 외부의 API (YouTube Data API v3)를 연동하며 고려한 점은?**
**답변:**
외부 API는 언제든 실패할 수 있으므로, 재시도(Retry) 패턴과 Fallback 메커니즘을 적용했습니다. `tenacity` 라이브러리를 통해 Exponential Backoff 방식의 3회 재시도를 적용했고, 그래도 실패할 경우 MOCK 데이터를 서빙하도록 처리해 장애 격리를 확보하였습니다. 

## 5. 프론트엔드와 백엔드의 통신
**Q5: 프론트엔드에서 API 토큰 인증이나 데이터 캐싱 전략은?**
**답변:**
지금은 개발 편의성 상 CORS 미들웨어를 모두 허용했지만, 실 운영 시 프론트엔드 도메인만 허용하고 세션 기반 또는 JWT 토큰을 사용하는 방식으로 수정해야 합니다. 프론트엔드 데이터 호출의 캐싱을 위해 LRU 캐시나 Redis 같은 인메모리 스토리지를 사용하는 방식이 부하를 막는 필수 최적화 단계입니다.

## 6. 테스트 주도 개발(TDD) 및 의존성
**Q6: 본 프로젝트에서 테스트는 어떻게 구성했고 단위 테스트의 이점은?**
**답변:**
`pytest`를 기반으로 Service 레이어에 대한 단위 테스트를 작성했습니다. Service 테스트 시 실제 DB Repository와 통신하지 않고 Mock Repository 객체를 주입하여 테스트의 속도와 독립성을 확보했습니다. 이를 통해 비즈니스 로직(점수 계산, 대화 전이 시스템)의 검증만 빠르게 할 수 있었습니다.

## 7. 대용량 트래픽에 대한 확장성 방안
**Q7: 갑자기 트래픽이 100배 증가한다면 어떤 부분을 수정할 것인가요?**
**답변:**
1. FastAPI 인스턴스를 늘리고 로드 밸런서(Nginx/ALB)를 도입합니다.
2. 현재 SQLite인 DB를 Master-Slave 구조의 DB로 마이그레이션하여 읽기/쓰기를 분리합니다.
3. YouTube API 호출 결과를 Redis 등에 캐싱( TTL 설정)하여 실 API 호출 횟수와 응답지연을 줄입니다.

## 8. 상태 기반 대화 모델링
**Q8: AI Chat 챗봇에서 세션 유지와 상태 관리는 어떻게 설계되었나?**
**답변:**
유저가 최초 응답을 받으면 UUID 형태의 session_id를 발급하고 이후 요청에 포함시킵니다. 내부적으로는 State Machine 형태(`ask_level`, `ask_topic`, `recommend`)를 적용하여 대화를 리드하고, DB에 상태값을 업데이트하며 컨텍스트를 보존합니다. LLM을 연동할 시 이 세션 관리를 프롬프트 체인 컨텍스트로 대체할 수 있도록 인터페이스를 분리했습니다.

## 9. FastAPI의 백그라운드 태스크 및 동시성
**Q9: FastAPI의 비동기 (async/await) 기능은 프로젝트 어디에 사용되었나?**
**답변:**
I/O 바운드가 큰 YouTube API 호출 부분에 `httpx.AsyncClient` 패키지를 활용해 Non-blocking async/await 기반으로 구현했습니다. 이로써 외부 네트워크 연동동안 해당 스레드가 멈추지 않고 다른 요청을 처리할 수 있어 서버 동시 처리량(Throughput)을 높였습니다.

## 10. 리팩토링의 경험치 (God Object 분리)
**Q10: 이번 리팩토링에서 가장 중요하게 개선된 부분은?**
**답변:**
God Object 구조(단일 `main.py` 파일 내 라우터, 비즈니스 로직, 인메모리 저장소의 혼재)의 분리입니다. 단일 책임 원칙(SRP)에 따라 파일을 라우터, 서비스, 리포지토리로 분리하였고, 이를 통해 소스 파일의 가독성 뿐만 아니라 다른 개발자와의 협업과 코드 리뷰 등 유지보수성이 가장 많이 개선되었습니다.
